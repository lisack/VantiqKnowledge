# Welcome to the Components Lab!

### Overview

You work in a science lab, writing Apps for data that comes in in real-
time. Even though the incoming data varies: temperature, speed,
populations, etc, you find yourself doing much the same with it in at least
part of each App you write and that is:
<br>
• Collect statistics on all of the data
• Filter out outliers
• Collect statistics on just the outliers
<br>
It's time to create an App Component to avoid some of this tedium in the
future!


By the time you complete this lab, you should:
<br>
• Be able to identify good candidates for App Components <br>
• Feel comfortable creating App Components<br>
• Be able to determine which aspect of your Components should be configurable<br>
• Be adept and consuming App Components<br>
<br>
#### Step 1. Verify what you have

a. There should be a package called “stats.transport” with an Event
Generator called “SpeedGen” and a Schema Type called
“KMperHour”<br>
b. You can ignore the lab.solution package, unless you’re stuck!

#### Step 2. Create A Service

a. Create a new Service from the Add menu; don’t forget to provide a package name, like “my.lab”<br>
b. Create a Visual Event Handler based on the stats.transport.KMperHour type <br>
	i. Set the EventStream to listen to the	/stats.transport/SpeedGen topic<br>
_Hint: Don't make a Public Event from the Interface tab, but instead go straight to the Implement Tab, and add a "Source" Visual Event Handler, setting the input to Topics, then the topic name_<br>
c. Save your App and run the stats.transport.SpeedGen generator to make sure your EventStream receives events<br>

#### Step 3. Create a Simple App
a. Using the Overview as a guide, create an App that performs the tasks listed in the Overview.<br>
b. Consider how long you want the sliding time window to be for collecting statistics; 5 minutes for testing purposes works<br>
_Hint1: A good “outlier” filter threshold for the incoming data might be anything over 130_ <br>
_Hint2: Your tasks should collect the statistics, but don’t be concerned about actually reading them – remember that an autogenerated procedure specific to the service is used for that. For now, we just want to collect them._<br>

#### Step 4. Create an App Component
a. Once you are satisfied that the App is working as expected, shift-click on the all the tasks that make up the functionality you want to replicate, and click on the Create Component button.<br>
_Hint: Be careful about accidentally moving the App tasks; if the “Save” button appears, the App will complain that you must save the App before creating a Component._<br>
b. In the name pane, be sure to include your package name, then the name of the new Component, e.g: “my.lab.statsComponent”<br>
c. A new window should pop up, and the title at the top left will be marked “App Component: <package.componentName>” <br>

#### Step 5. Configure the App Component

a. Think very carefully about what component properties will need to be alterable so that this component can be used for more than just highway speeds of cars, but also for lengths of things.<br>
b. Click on the “+” next to the properties that will need to be configurable, and give them intuitive names.
<br>
_Hint1: You may run into “Parameter Name Conflicts” if you used more than one of a kind of Activity Pattern to create your Component. Just choose “Create a New Parameter” and give it a distinct name._<br>

#### Step 6. Consume the App Component

a. Create a new Service and Internal Event Handler based on the stats.transport.CarLengthsM type<br>
b. Configure the inboundResourceId to /stats.transport/CarLengths so you can use the CarLengthsGen Event Generator<br>
c. Select and drag your new Component onto the canvas, connected to the EventStream<br>
d. With the Component selected, choose “Click to Edit” to see all the configurable properties:<br>
	i. The new event property for statistical analysis is: Length<br>
	ii. The filter expression to be evaluated is: Length > 4000<br>
e. Save the consumer App and start the CarLengthsGen Event Generator<br>
f. Assuming everything ran all right, if you want to see the two sets of statistics that are generated in the running App, drag the two “Get” service procedures to the Component Downstream and then drag a LogStream task under each of these to see statistics in the Log Messages window.<br>
h. Verify that the “all lengths” log messages have lower means than the “outlier lengths” log messages.


#### Congratulations!  

You have created your first App Component, made it flexible enough to work with any integer event value, and then used it as a Consumer.

From now on, as you build Apps, be looking for opportunities to make more App Components, to save yourself future time and effort!
