Agent Planning Patterns Within agents themselves, there's already built in logic for evaluating how best to fulfill a request.

Orchestrating agents, however, when they receive a request from a client, need an additional plan for where to send messages to one or more other agents in order to fulfill the request.

A LLM agent planning pattern is a way to prompt an LLM to achieve the most efficient plan for the purpose at hand.

There are multiple popular algorithms to implement the planning, but they all instruct the model to execute a sequence of tasks and report the results for the orchestrating agent.

Choose any planning pattern that fits your program's needs.

Each has its advantages and drawbacks.

The plan output needs to be a JSON structure of type IO dot Vantiq dot AI dot planning graph so the system can execute the plan.

We'll see more about this structure soon.

Now on to a concrete example.

This cake factory application receives real time events about machinery, health, ingredients, supply levels, incoming orders and so on.

And there are multiple agents specialized to deal with certain areas of responsibility like supplies, equipment, factory metrics, etcetera.

When a request from the dashboard comes in, the four men Orchestrators Coordinate Factory procedure first finds all the agents and their specializations.

Then it plugs the information about them into the planned procedure called Plan Request.

Plan Request generates a plan for determining which agent or agents should be directed to resolve the request.

In this example, when the plan is executed, the request goes to the equipment agent.

The equipment agent now uses its built in logic to determine which of its skills best fulfills the request and returns the results.
