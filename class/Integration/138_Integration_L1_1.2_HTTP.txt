We'll first define a few terms, then look at how Vantiq's API is structured for use before seeing practical examples of REST over HTTPAPI calls.

In the Vantiq application programming interface, resources are a class of object which establishes a certain set of characteristics about that class.

Operations are the actions that can be performed on resource instances like select, insert, and so on.

This diagram is a simpler rendition of the mechanisms by which outside systems are communicating with the Vantiq platform, whether it is a mobile app accessing the platform via a software development kit or a message protocol sending events to a Vantiq application, or even one part of the Vantiq platform communicating to another part.

Notice that almost all of this is taking place via a series of HTTP calls.

The only exceptions are cases when the communication channel needs to remain open in full duplex mode, like a continuous camera feed, database connection, etcetera.

Most web-based API's connect via HTTP over REST calls and Vantiq is no exception.

HTTP is a standard protocol and REST is a lightweight, stateless architectural style in common use for accessing application resources.

HTTP is considered a half duplex protocol, which is to say that the connection opens for each request individually and is considered closed after that.

This is why authentication information is required with each REST call.

All REST calls need to include the authorization information.

There are two types.
