Having your application system maintain different versions of the same rule is highly useful, especially as the system runs over time.

Let's say you have one central system that takes in events from multiple smaller applications, but the data some of them send has changed since you wrote the rule for it.

To version a rule, simply add a colon after the rule's name and the version number.

The system interprets the highest number to be the latest version and evaluates that first, followed by the next highest, and so on.

Be sure to be consistent in your numbering scheme or the system will interpret the versions incorrectly.

We've seen before that we can write rules to listen for a combination of events that all have to occur within a set period of time and have some relation to one another.

But that's not the only way.

Rules can listen for multiple events with the keyword or the rule can listen to any number of unrelated events, and there is no within delimiter because the events are treated separately.

This is very handy if multiple streams of data with the same structure are coming into the system and need to be handled the same way.

There are times, especially during development, when developers will want to turn off the rules from listening for and processing events.

To do this, just toggle off the activation checked box in the top right of the rule editor pane.

By default, when rules are deployed along with other resources to new nodes, they will be active.

But developers who want the rule to be inactive in some of the nodes where it is deployed just need to set an activation constraint.

The property should match a property on the nodes where the rule should be active.
